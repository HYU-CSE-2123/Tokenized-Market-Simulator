# ERC-20 기반 삼성전자 가격 추종 토큰 거래소 기획서

## 1. 프로젝트 개요

본 프로젝트는 Ethereum ERC-20 표준을 기반으로, 삼성전자 주가를 직접 토큰화하는 것이 아니라 **삼성전자 기준 가격을 추종하는 모의 price-tracking token**을 발행하고 거래할 수 있는 가상 거래소 시스템을 구현하는 것을 목표로 한다.

사용자는 Android 앱에서 모의 원화 자산을 이용해 삼성전자 가격 추종 토큰을 매수하거나 매도할 수 있으며, 백엔드는 가격 데이터, 주문 처리, 체결 내역, 포트폴리오, 실시간 가격 스트림을 관리한다. 블록체인 영역에서는 ERC-20 토큰, 가격 오라클, 거래 Vault 컨트랙트를 통해 매수·매도 정산 로직을 수행한다.

본 프로젝트의 토큰은 실제 삼성전자 주식, 배당권, 의결권, 상환권을 나타내지 않는다. 실제 금융상품이나 투자 서비스가 아닌, 블록체인·백엔드·모바일 클라이언트·실시간 거래 시스템 학습을 위한 모의 거래소 프로젝트이다.

---

## 2. 프로젝트 목적

본 프로젝트의 목적은 단순한 CRUD 서비스가 아니라, 금융 도메인의 핵심 흐름인 가격 조회, 주문, 체결, 잔고 반영, 실시간 데이터 스트리밍, 온체인 트랜잭션 상태 추적을 하나의 서비스로 구현하는 것이다.

구체적인 목표는 다음과 같다.

1. ERC-20 기반 모의 자산 발행 구조를 이해하고 구현한다.
2. 가격 오라클 기반 price-tracking token 거래 구조를 설계한다.
3. Spring Boot 백엔드에서 주문, 체결, 포트폴리오, 트랜잭션 상태를 관리한다.
4. WebSocket을 이용해 실시간 가격 및 체결 이벤트를 Android 앱에 전달한다.
5. Jetpack Compose 기반 Android 앱에서 실시간 금융 UI를 구현한다.
6. 향후 선물 거래, 레버리지, 청산 시스템으로 확장 가능한 도메인 구조를 설계한다.

## 3. 프로젝트 핵심 컨셉

### 3.1 Price-Tracking Token

본 프로젝트에서 발행하는 토큰은 실제 주식을 담보로 발행되는 stock-backed token이 아니다. 또한 실제 주식의 권리를 블록체인에 옮기는 tokenized stock도 아니다.

대신 특정 기준 가격을 오라클로부터 받아 해당 가격을 기준으로 매수·매도되는 **price-tracking token**이다.

예를 들어 삼성전자의 기준 가격이 75,000원이라고 가정하면, 사용자는 75,000 모의 원화를 지불하고 1개의 삼성전자 가격 추종 토큰을 매수할 수 있다. 이후 기준 가격이 80,000원으로 상승하면, 해당 토큰의 평가금액도 80,000 모의 원화로 계산된다.

### 3.2 실제 주식 토큰화와의 차이

| 구분          | 실제 주식 토큰화 | 본 프로젝트        |
| ----------- | --------- | ------------- |
| 실제 주식 보유    | 필요        | 없음            |
| 배당권         | 있을 수 있음   | 없음            |
| 의결권         | 있을 수 있음   | 없음            |
| 상환권         | 있을 수 있음   | 없음            |
| 실제 투자 기능    | 있음        | 없음            |
| 법적 금융상품 가능성 | 높음        | 모의 프로젝트로 제한   |
| 구현 목적       | 금융 서비스    | 기술 학습 및 포트폴리오 |

따라서 본 프로젝트에서는 “삼성전자 주식 토큰”이라는 표현을 사용하지 않고, **“삼성전자 가격 추종 토큰”** 또는 **“Samsung Electronics Price-Tracking Token”**이라는 표현을 사용한다.

### 6.1 Android 앱 역할

Android 앱은 사용자가 실제로 거래소를 사용하는 인터페이스 역할을 한다. 현재가, 차트, 매수·매도 화면, 포트폴리오, 체결 내역, 주문 상태를 제공한다.

### 6.2 Spring Boot 백엔드 역할

백엔드는 서비스의 중심 서버 역할을 한다. 사용자 인증, 주문 요청 검증, 가격 데이터 관리, 체결 내역 저장, 포트폴리오 계산, WebSocket 이벤트 전송, 블록체인 트랜잭션 상태 추적을 담당한다.

### 6.3 Blockchain 역할

블록체인 영역은 토큰 발행과 매수·매도 정산을 담당한다. ERC-20 토큰, 가격 오라클, 거래 Vault 컨트랙트를 통해 사용자의 모의 원화와 가격 추종 토큰의 이동을 기록한다.

### 6.4 Database 역할

PostgreSQL은 사용자 정보, 주문 내역, 체결 내역, 가격 기록, 트랜잭션 상태를 저장한다. 블록체인은 최종 정산 근거를 담당하고, DB는 서비스 화면에 필요한 조회와 상태 관리를 담당한다.

## 8. 가격 추종 방식

### 8.1 1차 구현 방식: 모의 가격 시뮬레이터

MVP 단계에서는 실제 삼성전자 실시간 시세를 사용하지 않고, 서버 내부의 가격 시뮬레이터를 사용한다.

초기 가격은 75,000원으로 설정하고, 일정 주기마다 랜덤한 변동률을 적용한다.

```text
초기 가격: 75,000
업데이트 주기: 1초 또는 3초
변동률: -0.3% ~ +0.3%
저장 위치: price_ticks 테이블
전송 방식: WebSocket broadcast
```

이 방식은 외부 API 의존성이 없고, 데모 환경에서 안정적으로 가격 변동을 보여줄 수 있다는 장점이 있다.

### 8.2 2차 구현 방식: 과거 시세 리플레이

2차 단계에서는 삼성전자 과거 시세 CSV 데이터를 준비하고, 이를 서버가 일정 주기로 재생하는 방식으로 확장한다.

```text
1. 삼성전자 과거 일봉 또는 분봉 데이터 준비
2. 백엔드가 CSV 데이터를 순차적으로 읽음
3. 일정 주기마다 다음 가격 tick을 발행
4. Android 앱은 이를 실시간 가격처럼 수신
5. 가격 변화에 따라 포트폴리오 평가금액 갱신
```

이 방식은 실제 시장 흐름과 유사한 데이터를 사용할 수 있어 시연 완성도가 높다.

### 8.3 3차 구현 방식: 실제 시세 API 연동

실제 API 연동은 마지막 단계에서 선택적으로 구현한다. 이 경우에도 실제 투자 서비스가 아니라 모의 거래 기준 가격으로만 사용한다.

## 9. 스마트 컨트랙트 설계

### 9.1 컨트랙트 목록

```text
contracts/
  MockKRW.sol
  SamsungPriceTrackingToken.sol
  PriceOracle.sol
  ExchangeVault.sol
```

### 9.2 MockKRW

모의 원화 ERC-20 토큰이다. 사용자는 faucet 기능을 통해 테스트용 mKRW를 지급받는다.

주요 기능은 다음과 같다.

```text
- ERC-20 기본 기능
- faucet()
- transfer()
- approve()
- transferFrom()
```

### 9.3 SamsungPriceTrackingToken

삼성전자 기준 가격을 추종하는 모의 ERC-20 토큰이다. 실제 삼성전자 주식을 나타내지 않으며, ExchangeVault 컨트랙트만 mint와 burn을 수행할 수 있다.

주요 기능은 다음과 같다.

```text
- ERC-20 기본 기능
- mint()
- burn()
- minter 설정
```

### 9.4 PriceOracle

현재 기준 가격을 저장하는 컨트랙트이다. 백엔드 또는 관리자 계정이 가격을 업데이트한다.

주요 기능은 다음과 같다.

```text
- priceE8 저장
- updatedAt 저장
- updatePrice()
```

가격은 정밀도 처리를 위해 `1e8` 단위로 저장한다.

예를 들어 75,000원은 다음과 같이 저장한다.

```text
75000 * 1e8
```

### 9.5 ExchangeVault

매수와 매도 정산을 담당하는 핵심 컨트랙트이다.

주요 기능은 다음과 같다.

```text
- buy(krwAmount)
- sell(tokenAmount)
- 수수료 계산
- 현재 오라클 가격 조회
- mKRW 수취
- mSEC mint
- mSEC burn
- mKRW 지급
```

## 11. 데이터베이스 설계

### 11.1 users

```sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    nickname VARCHAR(100) NOT NULL,
    wallet_address VARCHAR(255),
    created_at TIMESTAMP NOT NULL
);
```

### 11.2 assets

```sql
CREATE TABLE assets (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(20) UNIQUE NOT NULL,
    name VARCHAR(100) NOT NULL,
    contract_address VARCHAR(255),
    decimals INT NOT NULL
);
```

초기 데이터는 다음과 같다.

```text
mKRW / Mock Korean Won
mSEC / Samsung Electronics Price-Tracking Token
```

### 11.3 price_ticks

```sql
CREATE TABLE price_ticks (
    id BIGSERIAL PRIMARY KEY,
    symbol VARCHAR(20) NOT NULL,
    price DECIMAL(30, 8) NOT NULL,
    source VARCHAR(50) NOT NULL,
    created_at TIMESTAMP NOT NULL
);
```

### 11.4 orders

```sql
CREATE TABLE orders (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    symbol VARCHAR(20) NOT NULL,
    side VARCHAR(10) NOT NULL,
    order_type VARCHAR(20) NOT NULL,
    input_amount DECIMAL(30, 18) NOT NULL,
    expected_output_amount DECIMAL(30, 18),
    status VARCHAR(30) NOT NULL,
    tx_hash VARCHAR(255),
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);
```

주문 상태는 다음과 같이 정의한다.

```text
REQUESTED
PENDING_ONCHAIN
FILLED
FAILED
CANCELED
```

### 11.5 trades

```sql
CREATE TABLE trades (
    id BIGSERIAL PRIMARY KEY,
    order_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    symbol VARCHAR(20) NOT NULL,
    side VARCHAR(10) NOT NULL,
    price DECIMAL(30, 8) NOT NULL,
    base_amount DECIMAL(30, 18) NOT NULL,
    quote_amount DECIMAL(30, 18) NOT NULL,
    fee DECIMAL(30, 18) NOT NULL,
    tx_hash VARCHAR(255),
    created_at TIMESTAMP NOT NULL
);
```

### 11.6 blockchain_transactions

```sql
CREATE TABLE blockchain_transactions (
    id BIGSERIAL PRIMARY KEY,
    tx_hash VARCHAR(255) UNIQUE NOT NULL,
    type VARCHAR(50) NOT NULL,
    status VARCHAR(20) NOT NULL,
    block_number BIGINT,
    error_message TEXT,
    created_at TIMESTAMP NOT NULL,
    confirmed_at TIMESTAMP
);
```

## 18. 주요 기술적 난점

### 18.1 온체인 트랜잭션의 비동기성

블록체인 트랜잭션은 즉시 완료되지 않는다. 사용자가 매수 버튼을 눌렀다고 해서 곧바로 체결 완료로 처리하면 안 된다.

따라서 주문 상태를 다음과 같이 분리해야 한다.

```text
REQUESTED: 서버가 주문 요청을 받음
PENDING_ONCHAIN: 트랜잭션이 전송됨
FILLED: 블록체인에서 확정됨
FAILED: 트랜잭션 실패
```

### 18.2 DB 상태와 온체인 상태의 일관성

블록체인에서는 성공했지만 DB 업데이트가 실패하거나, DB에는 주문이 있는데 트랜잭션이 실패하는 문제가 생길 수 있다.

이를 해결하기 위해 `blockchain_transactions` 테이블을 따로 두고, 주기적으로 pending transaction을 재확인하는 reconciliation job을 구현한다.

### 18.3 가격 변동과 견적 유효성

사용자가 견적을 확인한 시점과 실제 매수 트랜잭션이 실행되는 시점의 가격이 달라질 수 있다.

MVP에서는 다음 정책을 사용한다.

```text
- 견적은 참고용으로 표시한다.
- 실제 체결 가격은 컨트랙트 실행 시점의 오라클 가격을 기준으로 한다.
- 앱에서는 “최종 체결 수량은 가격 변동에 따라 달라질 수 있음”을 표시한다.
```

2차 단계에서는 slippage tolerance를 추가할 수 있다.

### 18.4 Android 실시간 UI 상태 관리

가격, 주문 상태, 포트폴리오가 REST와 WebSocket 양쪽에서 갱신될 수 있다.

이를 위해 Android에서는 Repository에서 데이터 흐름을 통합하고, ViewModel에서는 StateFlow 기반으로 단일 UiState를 제공한다.

## 22. 포트폴리오 강조 포인트

이 프로젝트는 Android, 백엔드, 블록체인, 금융 도메인을 연결하는 종합 프로젝트이다.

포트폴리오에서는 다음 역량을 강조할 수 있다.

```text
1. ERC-20 기반 자산 모델링
2. 가격 오라클 기반 price-tracking token 구조 설계
3. Spring Boot와 스마트 컨트랙트 연동
4. 온체인 트랜잭션의 비동기 상태 관리
5. WebSocket 기반 실시간 가격 및 체결 스트리밍
6. Android Jetpack Compose 기반 실시간 금융 UI 구현
7. 주문, 체결, 포트폴리오 도메인 모델링
8. 향후 선물 거래로 확장 가능한 아키텍처 설계
```

이력서에는 다음과 같이 표현할 수 있다.

```text
Ethereum ERC-20 기반 삼성전자 가격 추종 토큰 거래소를 설계·구현했습니다. 실제 주식을 토큰화하는 방식이 아니라, 오라클 기준 가격을 따라 모의 매수·매도가 가능한 price-tracking token 구조로 설계했으며, Spring Boot 백엔드, PostgreSQL, WebSocket, Jetpack Compose Android 앱을 연동해 주문 생성, 온체인 트랜잭션 상태 추적, 체결 내역 저장, 포트폴리오 갱신, 실시간 가격 스트리밍을 구현했습니다.
```

## 23. 결론

본 프로젝트는 실제 금융 서비스를 구현하는 것이 아니라, 삼성전자 기준 가격을 추종하는 ERC-20 price-tracking token을 통해 모의 거래소 시스템을 구현하는 프로젝트이다.

초기 MVP는 오라클 가격 기반 즉시 매수·매도, 실시간 가격 스트림, 체결 내역, 포트폴리오 갱신에 집중한다. 이후 주문장 기반 거래, 과거 시세 리플레이, 실제 시세 API 연동, 선물 거래, 레버리지, 청산 시스템으로 확장할 수 있다.

이를 통해 단순 앱 개발을 넘어, 블록체인 자산 모델링, 백엔드 거래 도메인 설계, 실시간 데이터 처리, Android 금융 UI 구현 역량을 함께 보여줄 수 있다.
